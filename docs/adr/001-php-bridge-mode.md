# ADR-001: PHP Bridge Mode の採否

- **ステータス**: 検討中
- **日付**: 2026-03-07
- **関連Issue**: なし (GitHub Discussions で議論予定)

## コンテキスト

RustPressの最終目標は「全世界の全WordPressサイトの移行経路を確立する」ことである (MASTERPLAN Phase 11完了基準)。

WordPressエコシステムには59,000以上のPHPプラグインが存在し、多くのサイトがこれらに依存している。RustPressはPHP直接実行を行わない方針だが、これはプラグイン互換性の最大の障壁である。

この問題に対し「Hook発火時にPHP-FPMを呼び出してPHPプラグインをそのまま動かす (PHP Bridge Mode)」というアプローチが検討された。

## 検討した選択肢

### 案A: フックごとにPHP-FPM呼び出し

```
RustPress → [フック発火100回 × FastCGI往復] → Response
```

**結論: PHPより遅い。却下。**

- FastCGI 1往復: ~0.5-1ms
- 1ページあたりフック発火: 100-300回
- 追加オーバーヘッド: 50-300ms
- WordPress通常 (~200ms) より遅くなる

### 案B: リクエストごとに1回PHP-FPM呼び出し

```
RustPress → [1回のFastCGI: PHP側でWordPress bootstrap + 全フック実行] → Response
```

**性能: WordPressの約1.3倍速 (動的ページ)、25倍速 (キャッシュヒット)**

| 条件 | WordPress | PHP Bridge B |
|------|-----------|-------------|
| 動的ページ (キャッシュなし) | ~200ms | ~155ms |
| ページキャッシュヒット | ~50ms | ~2ms |
| 静的ファイル | ~10ms | ~0.1ms |
| 同時接続数 | ~500 | ~10,000+ |

- メリット: 全PHPプラグインがそのまま動く。移行障壁ゼロ。
- デメリット: PHP-FPMへの依存が残る。実質「WordPressの前にRustプロキシを置いた構成」。

### 案C: WordPress bootstrapなしでPHPプラグインフックだけ実行

```
RustPress → [PHPプラグインのコールバックだけ実行 (WP関数はRust側で処理)] → Response
```

**結論: 技術的に実現不可能。却下。**

PHPプラグインのコールバック関数はWordPress全体の実行コンテキスト内で動作する。具体的には:

1. **WordPress関数への依存**: `get_option()`, `get_post_meta()`, `is_singular()` 等のWP関数を呼ぶ
2. **グローバル状態への依存**: `$post`, `$wp_query`, `$wpdb` 等のグローバル変数を参照
3. **プラグイン自身のクラス/関数**: プラグインファイルが `require_once` されて初めて定義される
4. **Composerオートローディング**: 現代的なプラグインはPSR-4オートローダーに依存

これらを満たすにはWordPress bootstrapが必要 → 案Bと同じになる。

WP関数をRust側にブリッジする案 (案C') も検討したが、PHPコールバック内でWP関数を呼ぶたびにIPC往復が発生する「ピンポン地獄」となり、WordPressの5-7倍遅くなる。

```
1フック × 30 WP関数呼び出し × 0.5ms = 15ms/フック
PHPプラグインのフック数: 50-100個
合計: 750-1500ms ← WordPressの5-7倍遅い
```

### 案D: PHP Bridgeを採用し「脱PHP誘導」仕組みを組み込む

案Bをベースに、ユーザーが段階的にPHPプラグインをRustネイティブに置き換えるよう誘導する仕組みを組み込む。

```
管理画面に常時表示:
  RustPress Performance Score: 35/100
  PHP依存プラグイン: 18個（速度低下の原因）
  → Yoast SEO → rustpress-seo で代替可能 [ワンクリック移行]
  → CF7 → rustpress-forms で代替可能 [ワンクリック移行]
```

**問題: 「使っているうちにRustが出来上がる」は幻想。**

歴史的前例:
- **Wine** (WindowsアプリをLinuxで実行) → 20年経ってもLinuxネイティブアプリは増えなかった
- **JRuby** → CRuby本家が改善して出番減
- **React Native** → ブリッジが性能ボトルネックに。ネイティブ書き直し多発

動いているものを触るインセンティブがなく、Rustプラグインエコシステムが育たないリスクが高い。

### 案E: PHP Bridgeを捨て、AI変換一本に賭ける (現行方針)

PHPプラグインは動かさない。代わりに:
1. 主要プラグイン (上位50個) はRustで再開発
2. 中小プラグインはAI変換Webサービスで対応
3. 変換不可能なプラグインは代替を提案

## 未解決の論点

### WordPress互換性の範囲

PHPプラグインが動かない場合、WordPress「互換性」の意味は以下に限定される:

| レイヤー | 互換性 | 価値 |
|---------|--------|------|
| データベース (wp_posts等) | あり | **最大の差別化** — 既存WP DBにそのまま接続 |
| REST API (/wp-json/) | あり | 既存クライアント/モバイルアプリ対応 |
| URL構造 | あり | SEO維持 |
| プラグイン | なし | Rust再開発 or AI変換が必要 |
| テーマ | なし | Tera書き直しが必要 |

### DB互換は残すべきか？

プラグインもテーマも書き直すなら、WordPressのDB設計 (EAVパターン) に縛られる必要があるか？

**DB互換を捨てた場合:**
- 最適なスキーマ設計が可能 (EAV廃止、型安全)
- しかし「既存WP DBにそのまま接続」の訴求が消える
- Ghost, Strapi等との差別化が「Rustで書いた」だけになる

**DB互換を残した場合:**
- 8億サイトの移行先として唯一のポジション
- ゼロダウンタイム移行が可能
- EAVパターンの性能ペナルティを引き継ぐ

→ **DB互換は残すべき。これがRustPressの存在意義。**

### WP関数互換レイヤーの範囲

AI変換のターゲットとして:
- 現行案: PHP(WP API) → Rust(WP互換API) — 2000個のWP関数をRustで再実装
- 代替案: PHP(WP API) → Rust(RustPress独自API) — 最適設計のRust APIに直接変換

後者のほうが設計の自由度が高く、変換後のコード品質も良い可能性がある。

## 決定

**検討中。** コミュニティの意見を求めるためGitHub Discussionsに投稿する。

暫定方針:
1. **PHP Bridge Modeは採用しない** (案A/B/C いずれも問題あり)
2. **DB互換は維持する** (RustPressの存在意義)
3. **AI変換一本 + 主要プラグインRust再開発** (現行方針を継続)
4. **WP関数互換レイヤーの範囲は要検討** (AI変換のターゲットAPI設計と連動)

## 影響

- PHPプラグインをそのまま使いたいユーザーには移行障壁が残る
- AI変換サービスの品質がプロジェクト成否を大きく左右する
- 主要プラグインのRust再開発にリソースを集中する必要がある
- DB互換を維持することで、WPのEAVパターンに起因する性能課題を引き継ぐ
